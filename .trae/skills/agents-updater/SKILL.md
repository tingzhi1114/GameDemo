# Agents Updater —— AGENTS.md 自动更新 & 代码规范审查

## 核心职责

这个 Skill 负责两件大事：

### 1. AGENTS.md 自动进化（持续进行）

**无需用户批准**，在对话过程中主动捕获用户的编码偏好、习惯、纠正意见，并实时更新 AGENTS.md。

### 2. 代码规范审查（修改前 + 修改后）

在每次执行代码修改操作的前后，对照 AGENTS.md 中的规范进行检查，确保代码风格一致。

---

## 触发时机与行为

### 时机一：代码修改前（预检模式）

在用户要求修改代码，实际执行任何 **Write / SearchReplace / 创建文件** 操作**之前**：

1. **读取 AGENTS.md**，提取当前所有规范
2. **在脑海中预演**要写的代码结构和内容
3. **逐条核对规范**：
   - 命名规范是否遵循？（变量小蛇形、方法大驼峰、参数小驼峰）
   - 注释规范是否满足？（每个有含义的语句都有注释）
   - 代码复杂度是否合适？（优先简单直白，不追求高级语法）
   - 开发原则是否违反？（一类一文件、不报错原则等）
4. **如果不合规**：调整方案后再执行，确保一次通过

### 时机二：代码修改后（审查模式）

在每次 **Write / SearchReplace** 操作完成**之后**：

1. **立即读取**刚修改/创建的文件
2. **逐条对照 AGENTS.md 规范审查**：
   - 命名是否正确
   - 注释是否足够
   - 代码是否过于复杂
   - Unity 组件声明是否合理
   - 等等
3. **发现违规**：立即自动修复（无需询问用户）
4. **记录违规模式**：如果某种违规多次出现，考虑是否需要在 AGENTS.md 中追加新的规则来预防

### 时机三：偏好捕获（持续模式，贯穿整个对话）

**这是最核心的机制。** 在整个对话过程中，始终保持敏锐的感知：

#### 如何捕获显式偏好
用户明确说出的偏好：
- "我不喜欢用 var"
- "方法名应该用动词开头"
- 等等

→ 直接记录，更新到 AGENTS.md

#### 如何捕获隐式偏好（更难但更重要）
用户没有说出来，但从行为中可以推断的偏好：

- **用户自行修改代码**：当用户手动修改了你生成的代码，观察修改的内容——这是最强烈的隐式反馈信号
  - 比如你把一个变量命名为 `player_data`，用户改成了 `player_info` → 推断用户对命名有自己的偏好
  - 比如你把一段逻辑写成了一行 Lambda，用户改成了多行 if → 推断用户偏好可读性优先
- **用户接受/拒绝某些模式**：如果用户多次接受某类写法而不纠正，说明认可；如果某类写法用户总是要改，说明不认可
- **用户的提问方式**：用户问"这里为什么不用 enum？" → 说明用户偏好 enum 而非其他方案

#### 偏好 → AGENTS.md 的更新策略

| 情况 | 处理方式 |
|------|---------|
| 全新的规则/偏好 | 追加到 AGENTS.md 对应章节 |
| 与现有规则冲突 | 覆盖旧规则（用户最新的偏好为准） |
| 验证了现有规则 | 无需操作（用户行为确认了现有规则的正确性） |
| 用户明确推翻之前的规则 | 删除或修改旧规则 |

#### 更新 AGENTS.md 的原则

1. **保留基础结构**：不要删除文件开头的项目说明信息
2. **保持分类清晰**：新增内容放到对应的分类下（沟通风格、执行审批、代码风格、开发原则）
3. **语言一致**：AGENTS.md 用什么语言，就用什么语言写
4. **版本跟踪**：在文件末尾记录每次更新的日期和变更摘要

## 完整工作流程示例

### 场景：用户要求"帮我写一个 PlayerStats 类"

**Step 1 — 预检模式（修改前）：**
```
1. 读取 AGENTS.md
2. 检查到：类名用大驼峰 → "PlayerStats" ✓
   - 变量用小蛇形 → 计划用 current_hp, max_hp ✓
   - 每个语句都要注释 → 需要加注释 ✓
   - 一类一文件 → 单独文件 ✓
3. 确认方案合规，执行
```

**Step 2 — 执行写入**

**Step 3 — 审查模式（修改后）：**
```
1. 读取刚创建的文件
2. 逐条检查...
3. 发现问题：某个方法没有注释 → 立即补上
```

**Step 4 — 持续偏好捕获：**
```
用户说："我不喜欢 current_hp 这种缩写，用 current_health 更好"
→ 这是一个命名偏好的反馈
→ 更新 AGENTS.md：在命名规范中追加"不使用缩写，即使用全拼"
```

## 注意事项

- **无需询问**：更新 AGENTS.md 和修复代码违规都不需要向用户请示
- **保持克制**：不要过度推断用户的偏好。某个模式出现 **2 次以上** 才视为稳定的偏好
- **保留上下文**：在 AGENTS.md 中更新规则时，如果可能，简要说明原因（方便未来回顾）
- **不要破坏项目**：在修改代码时，注意不要引入编译错误
- **尊重用户覆盖**：如果用户手动修改了你写的代码，以用户的版本为准，并从中学习
