# Unity项目规范（必须遵循以下所有规范）

## 一、沟通风格

- 复杂问题简要解释 + 示例。
- 善于独立思考，对于用户的提议，常能想出可能存在的大问题，并告知用户可行性，不完全迎合用户
- 帮用户解决问题后，再简单的问题也需要指出你发现的问题、背后的原理、你是怎么做的。
- 对于不太清楚的需求，积极向用户进行提问（如果你已有建议，最好给与问题的对应选项），保证完全理解用户需求再行动
- **不要一次性给出完整方案/框架，每次只说当前这一步需要做什么**，像搭积木一样一块一块来。用户说"step3"只给step3的内容，不给step4、step5的预告。

## 二、执行与审批原则

- 默认只讨论不行动  ，用户明确指令才可执行代码修改、文件写入等操作。
- 所有非【只读】性质的行为，必须在最后向用户发送申请，待用户同意后方可进行，一次申请可包括多个操作。
- 单次批准仅当次有效  ，连续行动需重新确认。
- 只读查看文件无需批准  ，可自由浏览代码理解项目。
- 除非新文件和Unity强相关，否则文件放在Another目录中的子目录（若无同类内容则创建新目录）  。
- 如果产生新文件是为了AI（你）确认某些事情（例如你的短期行动方针、临时截图等），则无需保存该文件（删除该文件），除非用户明确要求保留该文件  。
- 使用中文思考，且用中文回复用户。
- 列出计划清单，在进行工作前，列出清单给与用户确认，若任务进行中发现无法实现清单的某一项，则立刻停止，并思考后续修改方案
- 每次执行任务时，可能用户自行进行了修改（保证信息的实时性），你必须以用户的修改为基准去执行任务，而非重新纠正用户修改过的文件（除非你认为用户的修改有严重错误或逻辑问题，则你可以提出建议，确认用户是否坚持这样改，否则直接默认以用户修改过的文件为基准）
- **写入前检查文件修改时间**：每次执行 Write/SearchReplace 前，先检查相关脚本的最后修改时间。如果发现文件比你的最终修改时间更新，说明用户已手动修改过代码，必须先重新读取再执行操作，避免覆盖用户的改动。
- **引用代码时使用超链接**：每次提及代码修改或文件位置时，使用 markdown 超链接 `[file_path#Lxx-Lxx](file:///absolute/path/to/file#Lxx-Lxx)` 格式精确定位到代码块。
- **写完代码后自动刷新 Unity 编辑器**：每次通过 Write/SearchReplace 等工具修改代码后，使用 Unity MCP 工具触发编辑器刷新，避免用户需要手动点击 Unity 重新加载。

## 三、代码风格

### 命名规范

- 变量  、常量等小结构使用小蛇形，如 `current_hp``、`energy\_dict\`
- 方法、枚举、类等大结  使用大驼峰，如 `GenerateTurnResources()`、`TakeDamage()`
- 参数使用小驼峰，如`currentHp`
- 不使用大蛇形
- **代码中不允许出现中文字符（注释除外）**。枚举值、字符串字面量等均使用英文。
- **枚举值需要同行注释**：每个枚举值在同一行末尾添加 `// 中文说明` 注释。
- **构造函数必须使用 this.xx = xx 形式**，不接受无参构造函数（除非有特殊理由）。外部构造时必须传入所有参数，不允许有默认值。
  ```csharp
  public CharacterData(string name, int gender, int age, float strength, float agility, float wit, float charm, float physique, float luck)
  {
      this.name = name;
      this.gender = gender;
      // ...
  }
  ```

- **构造器调用必须使用命名参数（named arguments）**：类似于 `id: 1, name: "张三"` 的形式，让每个参数都明确对应到字段，避免裸数字导致可读性差。

### 注释规范

- 注释详细到每个有含义的语句。（即每一个代码块，比如一个循环、一个方法体、一个if判断等）
- 大块的区域注释（如类、结构体、方法体等）使用///summary///，小区域注释（如某一行语句）使用//单行注释
- 注释内容以贴合项目背景为前提，尽量直白地说人话。
- 追求高可读性，让人一眼看懂这段代码在项目里代表什么。

### 代码复杂度

- 若某功能的实现，用简单语法与高端语法的代码量及复杂性、后期可维护性、代码可读性都差不多，则优先使用直白的简单语法。（即不必为了节省代码行数，而去写高级的复合代码，例如能用两个if来写的功能，就不用用一行代码去解决）
- 不要为了精简一两行代码使用高级的语法，而是优先通俗易懂的写法。
- 尽量内联代码，避免多层封装，保持代码简洁。对于非通用的方法，如果该方法体不大，就几行，则可以考虑直接内联到调用处。
- **禁止使用高级语法**：包括但不限于 switch 表达式、Lambda 表达式（除非极简单）、LINQ 查询语法、三目运算符嵌套等。一律使用朴素的 if-else、foreach/for 循环。

## 四、开发原则

- **AGENTS.md 只收录通用的编码规范**（命名、注释、语法风格、文件组织等），不收录当前项目具体实现的数据类型约定（如某个字段是 int 还是 float、有没有上限等）。具体实现的类型选择是随时可变的，不视为规范。
- 一类一文件  ：每个类单独一个文件，不堆在一起。枚举单独放在一个全局枚举文件 `GlobalEnums.cs` 中，所有枚举都放在此处。枚举结尾统一使用Enum
- 不报错原则  ：分步实现时，先写空方法占位，确保编译通过，再逐步填充逻辑。
- 改完先自查  ：修改后主动检查编译报错，确认无错再向用户提交结果。
- 修改透明  ：每次行动后告知用户修改了哪些文件。
- 完成某次行动后，在最后再重新思考，查看你刚执行的操作，判断逻辑是否正确，是否有错误，若有错误则修改好，然后再次重新审视，直至无错误为止。
- Unity组件  ：在代码中声明Unity组件时，尽量选择具有一定含义的父物体，而不是直接选择单一组件，可通过父物体再在代码中找到目标组件。避免选择含义太过广泛例如Canvas，也避免选择含义太过狭小例如Text作为申明挂载的物体。而是选择类似Timeline\_Zone这种有明确含义的组件。
- Unity中使用TMP而非传统Text
- 对于类，判断是否适合单例模式设计条件，若符合，可以考虑使用单例模式

### XXXDictionary 数据容器模式

- 所有充当"数据库"角色的 XXXDictionary 类（角色字典、地点字典等），**硬编码初始数据统一在构造器中使用集合初始化器语法 `new Dictionary<T>() { { key, value }, ... }`**，不允许多行逐条赋值。
- 遵循固定结构：单例 + `private Dictionary` + `private 构造器` + `Add/Get/Remove/GetAll` 四个公开方法。
- 构造器必须是 private，杜绝外部 new。
